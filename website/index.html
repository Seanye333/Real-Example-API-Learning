Lesson 01

GET Requests

GET requests are used to retrieve data from a server. Think of it like asking a question — you're requesting information.

Real World Analogy

Imagine you walk into a library and ask the librarian: "Can I see the book with ID #1?"
The librarian (server) finds the book and hands it to you (response). That's exactly what a GET request does.

How It Works (Step by Step)

  1. Your code sends an HTTP GET request to a URL (the server's address)
  2. The server receives your request and processes it
  3. The server sends back a response containing:
    • A status code (200 = success, 404 = not found, etc.)
    • Headers (metadata like content type, size)
    • A body (the actual data, usually JSON)
  4. Your code reads the response and uses the data

What is JSON?

JSON (JavaScript Object Notation) is a text format for sending data. It looks almost identical to a Python dictionary:

{"name": "John", "age": 30, "city": "New York"}

APIs use JSON because it's lightweight, readable, and universal — every programming language can read it.

What is the requests library?

requests is a Python library that makes HTTP calls easy. Install with:

pip install requests
Example 1

Simple GET — Fetch a Single User

import requests

# Send a GET request to fetch user #1
response = requests.get("https://jsonplaceholder.typicode.com/users/1")

# Check if it worked
print(response.status_code)  # 200 = success

# Convert JSON response into a Python dictionary
user = response.json()

print(user['name'])              # "Leanne Graham"
print(user['email'])             # "Sincere@april.biz"
print(user['address']['city'])  # "Gwenborough" (nested data!)

The Response Object

response.status_codeHTTP status code (200, 404, 500...)
response.json()Body parsed as a Python dict/list
response.textBody as a raw string
response.headersMetadata about the response
response.urlThe URL that was actually requested
response.okTrue if status code is 200-299
Example 2

GET a List — Fetch All Posts

# When you request /posts (no ID), the API returns ALL posts as a list
response = requests.get("https://jsonplaceholder.typicode.com/posts")
posts = response.json()  # Returns a LIST of dictionaries

print(f"Total posts: {len(posts)}")  # 100

# Loop through the first 3 posts
for post in posts[:3]:  # [:3] = first 3 items (Python slicing)
    print(f"  Post #{post['id']}: {post['title'][:50]}...")
Example 3

GET with Query Parameters — Filter Results

# Query parameters filter results. They go after ? in the URL:
# https://jsonplaceholder.typicode.com/posts?userId=1
#                                            ^^^^^^^^ query param

# Better way: use a params dict — requests builds the URL for you
params = {"userId": 1}
response = requests.get(
    "https://jsonplaceholder.typicode.com/posts",
    params=params,
)
filtered = response.json()

print(f"Posts by user 1: {len(filtered)}")  # 10
print(response.url)  # See the full URL that was sent
Example 4

Inspecting Response Headers

What Are Headers?

Headers are metadata sent alongside the response — like the label on a package. The package contents = the body (your data). The label = headers (info about the data).

response = requests.get("https://jsonplaceholder.typicode.com/posts/1")

print(response.headers['Content-Type'])  # application/json; charset=utf-8
print(len(response.content))              # Response size in bytes
print(response.encoding)                  # utf-8
Example 5

Handling Errors Gracefully

Why Handle Errors?

APIs can fail: resource doesn't exist (404), server is down (500), you're not authorized (401). Good code always handles these instead of crashing!

# Method 1: Check status code manually
response = requests.get("https://jsonplaceholder.typicode.com/posts/99999")

if response.status_code == 200:
    print("Success!")
else:
    print(f"Error! Status: {response.status_code}")


# Method 2: Use try/except with raise_for_status()
try:
    response = requests.get("https://jsonplaceholder.typicode.com/users/99999")
    response.raise_for_status()  # Throws exception if 4xx or 5xx
except requests.exceptions.HTTPError as e:
    print(f"HTTP error: {e}")
except requests.exceptions.ConnectionError:
    print("Can't reach the server!")
except requests.exceptions.Timeout:
    print("Server took too long!")

Lesson 02

POST Requests

POST requests are used to send/create new data on a server. Think of it like filling out a form and submitting it.

Real World Analogy

Imagine you're at a restaurant and fill out an order form: Name: "Burger", Extras: "Cheese", Quantity: 2.
You hand the form (your data) to the waiter (server). The waiter processes it and hands you a receipt (response) with an order number (new ID).

GET vs POST

GET"Give me data" — you send nothing, server sends data back
POST"Here's new data" — you send data, server creates something

GET is like reading a book. POST is like writing a new page in the book.

What is a Request Body?

When you POST, you include a "body" — the data you're sending. Usually JSON:

{"title": "My Post", "body": "Hello!", "userId": 1}
Example 1

Simple POST — Create a New Post

# Prepare the data as a Python dictionary
new_post = {
    "title": "My First API Post",
    "body": "Learning POST requests!",
    "userId": 1,
}

# json= does TWO things automatically:
#   1. Converts your dict to a JSON string
#   2. Sets the Content-Type header to "application/json"
response = requests.post(
    "https://jsonplaceholder.typicode.com/posts",
    json=new_post,
)

print(response.status_code)       # 201 = Created
print(response.json()['id'])      # Server assigned ID: 101

Status 201 vs 200

200 = "OK" (general success, used for GET)
201 = "Created" (a new resource was made, used for POST)

Example 2

POST with Manual Headers

import json

headers = {
    "Content-Type": "application/json; charset=UTF-8",
}

payload = {"title": "Manual Post", "body": "Test", "userId": 2}

response = requests.post(
    "https://jsonplaceholder.typicode.com/posts",
    data=json.dumps(payload),  # Manually convert dict to JSON string
    headers=headers,           # Manually set Content-Type
)

Three Ways to Send Data

json=my_dictAutomatic JSON (preferred)
data=json.dumps(my_dict) + headersManual JSON (when you need control)
data=my_dictForm-encoded data (like HTML forms)
Example 3

POST Form Data

What is Form Data?

When you submit a form on a website, the browser sends it as form-encoded, not JSON.

Form-encoded: title=Hello&body=World&userId=1
JSON: {"title": "Hello", "body": "World", "userId": 1}

Most modern APIs use JSON. Some older or file-upload APIs use form data.

form_data = {
    "title": "Form Post",
    "body": "Sent as form-encoded",
    "userId": 3,
}

# data= WITHOUT json.dumps sends as form-encoded
response = requests.post(
    "https://jsonplaceholder.typicode.com/posts",
    data=form_data,  # Form data, NOT JSON
)
Example 4

Handling POST Errors

Why Error Handling Matters More for POST

GET just reads data — if it fails, nothing bad happens.
POST creates data — if it fails mid-way, you might get duplicates or partial data. Always handle errors!

try:
    response = requests.post(
        "https://jsonplaceholder.typicode.com/posts",
        json={"title": "Test"},
        timeout=5,  # Give up after 5 seconds
    )
    response.raise_for_status()
    print(f"Created: {response.json()['id']}")
except requests.exceptions.Timeout:
    print("Timed out!")
except requests.exceptions.HTTPError as e:
    print(f"HTTP Error: {e}")
except requests.exceptions.ConnectionError:
    print("Can't connect!")

Lesson 03

PUT, PATCH & DELETE

The remaining HTTP methods for updating and deleting resources.

Real World Analogy

Think of a contact in your phone: Name: John, Phone: 555-1234, Email: john@email.com

  • PUT: Delete the entire contact and re-enter everything from scratch. Forget the email? It's gone.
  • PATCH: Tap on just the phone number and change it. Name and email stay exactly as they were.
  • DELETE: Delete the contact entirely.

CRUD Operations

CreatePOSTPOST /posts
ReadGETGET /posts/1
UpdatePUT / PATCHPATCH /posts/1
DeleteDELETEDELETE /posts/1

The URL identifies what resource. The HTTP method identifies what action.

Example 1

PUT — Full Replace

Warning

PUT replaces everything. If you forget a field, it gets erased!

# Must send ALL fields — missing ones get erased!
updated = {
    "id": 1,
    "title": "New Title",
    "body": "New body",
    "userId": 1,
}

response = requests.put(
    "https://jsonplaceholder.typicode.com/posts/1",
    json=updated,
)
print(response.status_code)  # 200
print(response.json())       # Full replaced object
Example 2

PATCH — Partial Update

# Only send what you want to change — rest stays the same
response = requests.patch(
    "https://jsonplaceholder.typicode.com/posts/1",
    json={"title": "Only Title Changed"},
)

result = response.json()
print(result['title'])   # "Only Title Changed" — updated!
print(result['body'])    # Still has original body — untouched!
print(result['userId'])  # Still 1 — untouched!
Example 3

DELETE — Remove a Resource

# Just the URL — no body needed
response = requests.delete(
    "https://jsonplaceholder.typicode.com/posts/1"
)

print(response.status_code)  # 200
print(response.json())       # {} (empty — it's gone)

In real APIs, DELETE is permanent! There's usually no undo.

Example 4

PUT vs PATCH — Side by Side

PUT (Full Replace)

  • Send ALL fields
  • Missing fields get erased
  • Like rewriting an entire document
  • Use when you have the complete object

PATCH (Partial Update)

  • Send ONLY changed fields
  • Other fields stay untouched
  • Like editing one paragraph
  • Use when tweaking a few fields
Example 5

Complete CRUD Cycle

BASE = "https://jsonplaceholder.typicode.com/posts"

# 1. CREATE
r = requests.post(BASE, json={"title": "New", "body": "Hi", "userId": 1})
print(f"Created ID: {r.json()['id']}")

# 2. READ
r = requests.get(f"{BASE}/1")
print(f"Title: {r.json()['title']}")

# 3. UPDATE
r = requests.patch(f"{BASE}/1", json={"title": "Updated!"})
print(f"Updated: {r.json()['title']}")

# 4. DELETE
r = requests.delete(f"{BASE}/1")
print(f"Deleted! Status: {r.status_code}")
